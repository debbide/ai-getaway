# 项目优化建议

检查时间：2026-06-03

## 检查范围

- 后端：`config/`、`database/`、`router/`、`middleware/`、`controller/`、`service/`、`upstream/`、`model/`
- 前端：`frontend/src/`、`frontend/vite.config.js`、`frontend/package.json`
- 文档与配置：`README.md`、`.env.example`、`.gitignore`、`部署流程.md`

## 验证结果

- 后端测试通过：`go test ./...`
- 前端构建通过：在 `frontend` 目录执行 `node node_modules\vite\bin\vite.js build`
- 前端构建提示主包较大：`dist/assets/index-*.js` 约 2512.85 kB，gzip 后约 835.04 kB；`dist/assets/index-*.css` 约 558.51 kB，gzip 后约 85.97 kB

## P0：安全与生产配置

### 1. 移除部署文档中的真实或近似真实敏感值

`部署流程.md` 中出现了 `REDIS_PASSWORD`、`JWT_SECRET`、`DEFAULT_ADMIN_PASSWORD` 的具体示例值。即使只是示例，也容易被复制到生产环境，或者造成历史泄露风险。

建议：

- 将具体值替换为占位符，例如 `请替换为强随机密码`。
- 如果这些值曾经真实使用过，立即轮换 Redis 密码、JWT Secret、默认管理员密码。
- 增加文档说明：生产环境不得复用示例密钥。

### 2. 拆分 JWT 签名密钥与数据加密密钥

当前 `JWT_SECRET` 同时用于 JWT 签名、API Key 加密和 OAuth state HMAC。一个密钥承担多个用途会放大泄露影响，也不利于后续轮换。

建议：

- 新增 `API_KEY_ENCRYPTION_SECRET` 和 `OAUTH_STATE_SECRET`。
- 保留 `JWT_SECRET` 只用于 JWT。
- 做兼容迁移：旧密钥可短期解密，重新展示或轮换 API Key 时写入新密钥加密结果。

### 3. 生产环境禁止隐式使用 JWT Secret 作为集群内部令牌

`CLUSTER_INTERNAL_TOKEN` 默认回退到 `JWT_SECRET`。这会把用户会话签名密钥暴露到内部服务调用配置中。

建议：

- 生产环境要求显式配置 `CLUSTER_INTERNAL_TOKEN`。
- 在 `Config.Validate()` 中校验生产环境下 `CLUSTER_INTERNAL_TOKEN` 不为空，且不能等于 `JWT_SECRET`。

## P1：代理层稳定性与资源控制

### 4. 让 `UPSTREAM_TIMEOUT` 真正作用到反向代理

配置中已经有 `UPSTREAM_TIMEOUT`，但 `upstream.ProxyHandler` 使用的是 `httputil.NewSingleHostReverseProxy` 的默认 transport，没有看到显式超时、连接池、TLS 握手超时或响应头超时配置。

建议：

- 解析 `cfg.UpstreamTimeout` 后注入代理层。
- 配置自定义 `http.Transport`，至少包含 `DialContext`、`TLSHandshakeTimeout`、`ResponseHeaderTimeout`、`IdleConnTimeout`、`MaxIdleConns`、`MaxIdleConnsPerHost`。
- 对 SSE 请求保留较长读取窗口，但普通请求应设置合理超时。

### 5. 给代理请求体读取增加大小上限

`parseRequestInfo()` 和 `applyDynamicOutputLimit()` 会读取完整请求体。大请求可能造成内存压力，尤其是用户通过 `/v1/*` 直接透传。

建议：

- 新增 `MAX_PROXY_BODY_BYTES`，默认例如 10 MB 或按业务模型能力设置。
- 对超限请求返回 413。
- 对无需解析的 endpoint 尽量避免重复读取请求体。

### 6. 优化 SSE 用量统计的内存占用

`usageStreamReadCloser` 当前会把流式响应累积到 buffer，在 EOF 后再解析。长文本或长时间流式响应会持续占用内存。

建议：

- 改成边读边按行解析 `data:` 事件，只保留必要的 usage 片段。
- 对无法解析 usage 的流式响应，保留最终兜底计费逻辑。
- 给 SSE buffer 增加最大容量保护，超过后停止缓存但继续透传。

### 7. 将 API Key 分钟限流改为可配置

`allowAPIKey()` 固定为每分钟 120 次。不同套餐、公开池和专属上游的合理限流不同。

建议：

- 增加全局配置 `API_KEY_RATE_LIMIT_PER_MINUTE`。
- 后续可扩展到套餐级或用户级限流字段。
- Redis 不可用时当前会放行，可考虑在生产环境支持 `fail-open` / `fail-closed` 配置。

## P1：数据库与迁移

### 8. 将生产迁移从 `AutoMigrate` 逐步切到版本化迁移

当前启动时执行 `AutoMigrate`，并包含删除旧字段逻辑。生产环境中自动结构变更可控性较弱，回滚和审计也不方便。

建议：

- 引入版本化迁移目录，例如 `migrations/`。
- 启动时只执行必要且可幂等的轻量初始化。
- 删除字段、索引调整、字段类型变更放到显式迁移脚本中。

### 9. 补充关键查询索引复核

模型里已有不少 `index`，但一些高频后台列表、用量统计和支付状态查询建议按真实查询条件复核联合索引。

建议优先检查：

- `orders(user_id, status, created_at)`
- `api_logs(user_id, created_at)`、`api_logs(api_key_id, created_at)`、`api_logs(model, created_at)`
- `quota_reservations(user_id, status, completed_at)`
- `channel_monitor_records(channel_monitor_id, checked_at)`

## P1：前端体积与可维护性

### 10. 拆分前端大组件和路由级代码

当前文件体量较集中：`AdminPanel.vue` 约 217 KB，`style.css` 约 199 KB，`App.vue` 约 57 KB。构建后主 JS 约 2.5 MB，已经触发 Vite chunk 警告。

建议：

- 将后台页面按功能拆分为用户、订单、套餐、渠道、模型、文档、公告、日志等子组件。
- 用动态 `import()` 做页面级懒加载。
- 在 `vite.config.js` 配置 `manualChunks`，将 `element-plus`、编辑器、Markdown 相关库拆到独立 chunk。

### 11. 整理全局 CSS

`frontend/src/style.css` 接近 200 KB，长期会增加样式覆盖和回归风险。

建议：

- 将页面私有样式移回对应 Vue SFC。
- 保留全局变量、基础排版、Element Plus 主题覆盖在全局 CSS。
- 抽出可复用的布局类和状态类，减少重复选择器。

### 12. 评估富文本编辑器按需加载

`@wangeditor/*`、`marked`、`dompurify`、`turndown` 只在文档、公告或编辑预览场景高频使用，但会进入主包。

建议：

- 管理端编辑器组件使用懒加载。
- Markdown 渲染工具保留安全净化，但可以拆入文档/公告相关 chunk。

## P2：测试覆盖

### 13. 补充配置校验测试

`config.Config.Validate()` 承担生产安全配置拦截，建议补测试覆盖。

建议覆盖：

- 生产环境拒绝默认 `JWT_SECRET`。
- 生产环境拒绝默认管理员密码。
- 生产环境拒绝通配 `ALLOWED_ORIGINS`。
- 后续新增 `CLUSTER_INTERNAL_TOKEN` 独立校验后覆盖其规则。

### 14. 补充路由权限边界测试

当前已有 controller、middleware、service、upstream 的部分测试，但 admin 路由和普通用户路由边界建议系统化覆盖。

建议覆盖：

- 未登录访问 `/api/admin/*` 返回 401。
- 普通用户访问 `/api/admin/*` 返回 403。
- API Key 禁用、用户禁用、订阅过期、协议不支持等代理前置拦截。

### 15. 补充代理异常和流式场景测试

代理层是核心业务路径，建议继续增强测试。

建议覆盖：

- 上游非 2xx 时取消额度预占。
- 上游连接失败时取消额度预占。
- SSE usage 分片跨 read 边界时仍能正确统计。
- 请求体超限时返回 413。

## P2：文档与工程化

### 16. 修正 `.gitignore` 对锁文件的处理

当前 `.gitignore` 忽略了 `go.sum` 和 `frontend/package-lock.json`。但仓库中已有 `go.sum`，AGENTS 也提到 `npm install` 应从 `package-lock.json` 安装。锁文件通常应提交以保证可复现构建。

建议：

- 不再忽略 `go.sum`。
- 如果前端使用 npm，提交 `frontend/package-lock.json`。
- 如果确实不希望提交锁文件，需要同步修改 AGENTS 和 README 的安装说明。

### 17. 给环境变量补一份分层说明

`.env.example` 已列出变量，但可以进一步标注必填、生产必填、默认值风险和是否可轮换。

建议：

- 新增 `md/环境变量说明.md` 或补充 README。
- 对密钥类配置标注生成方式和轮换影响。
- 对 `RUN_BACKGROUND_JOBS`、`CLUSTER_MODE`、`INSTANCE_ID` 补充多实例部署说明。

### 18. 增加 CI 检查脚本

当前本地验证命令明确，但缺少统一脚本。

建议：

- 后端：`go test ./...`
- 前端：`node node_modules\vite\bin\vite.js build`
- 可加一个 Windows 批处理或 PowerShell 脚本，避免 npm 全局入口缺失导致误判。

## 建议实施顺序

1. 先处理部署文档敏感值、密钥拆分、集群令牌独立配置。
2. 再处理代理超时、请求体大小限制、SSE 内存优化。
3. 然后做前端拆包和大组件拆分，降低首屏资源体积。
4. 最后补齐迁移体系、索引复核和测试覆盖。
