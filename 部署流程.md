# Windows 本地构建 + Debian 1Panel 部署流程

本文档适用于：代码在 Windows 本地构建，Linux 服务器只负责运行部署。服务器系统为 Debian 11/12，并使用 1Panel 管理 OpenResty、MySQL/MariaDB、Redis 和站点反向代理。

示例域名写作 `your-domain.com`，请替换为你的真实域名。

## 1. 部署架构

```text
用户浏览器
  |
  | HTTPS
  v
1Panel / OpenResty
  |-- 静态文件：/opt/ai-getaway/frontend/dist
  |-- /api/  -> http://127.0.0.1:8080/api/
  |-- /v1/   -> http://127.0.0.1:8080/v1/
  |
  v
Go 后端 systemd 服务
  |
  |-- MySQL/MariaDB
  |-- Redis
```

构建发生在 Windows 本地；Debian 服务器不需要安装 Go、Node.js、npm。

多服务器部署时建议分成：

```text
主包服务器
  |-- 1Panel / OpenResty / HTTPS / 前端 dist
  |-- Go 后端节点 backend-1
  |-- 可选：MariaDB 主库、Redis 主节点或它们的访问入口

从包服务器
  |-- Go 后端节点 backend-2、backend-3 ...
  |-- 不部署前端，不创建网站
  |-- 连接同一套 MariaDB 和 Redis
```

所有用户仍然访问同一个域名。主入口 OpenResty 或云负载均衡把请求转发到健康的后端节点。单机部署时可以完全忽略从包步骤，按主包部署即可。

## 2. Windows 本地构建

以下命令在 Windows 的项目目录中执行，例如：

```powershell
cd D:\python\ai-getaway
```

### 2.1 构建后端 Linux 二进制

在 PowerShell 中执行交叉编译：

```powershell
$env:GOOS="linux"
$env:GOARCH="amd64"
$env:CGO_ENABLED="0"
go build -o ai-getaway-linux-amd64 .
```

构建完成后会得到：

```text
D:\python\ai-getaway\ai-getaway-linux-amd64
```

如果你的服务器是 ARM64，例如部分 ARM 云服务器，改用：

```powershell
$env:GOOS="linux"
$env:GOARCH="arm64"
$env:CGO_ENABLED="0"
go build -o ai-getaway-linux-arm64 .
```

### 2.2 构建前端

```powershell
cd D:\python\ai-getaway\frontend
npm install
npm run build
```

如果当前 Windows 的全局 `npm` 损坏，也可以直接使用项目本地 Vite：

```powershell
node node_modules\vite\bin\vite.js build
```

构建完成后会得到：

```text
D:\python\ai-getaway\frontend\dist
```

### 2.3 准备上传文件

如果只部署一台服务器，按主包方式上传即可。主包包含前端、后端和站点入口；从包用于第二台及更多后端服务器，只需要后端二进制和 `.env`。

主包建议上传这些内容：

```text
ai-getaway-linux-amd64
.env.example
frontend/dist/
```

主包运行实际只需要：

```text
/opt/ai-getaway/ai-getaway
/opt/ai-getaway/.env
/opt/ai-getaway/frontend/dist/
```

从包建议上传这些内容：

```text
ai-getaway-linux-amd64
.env.example
```

从包运行实际只需要：

```text
/opt/ai-getaway/ai-getaway
/opt/ai-getaway/.env
```

从包不需要上传 `frontend/dist/`，也不需要在从服务器上创建 1Panel 网站。它只作为后端节点接收主入口或负载均衡转发过来的 `/api/`、`/v1/`、`/messages` 请求。

## 3. 安装 1Panel

以下命令在 Debian 服务器执行。

```bash
apt update
apt upgrade -y
apt install -y curl wget vim unzip ca-certificates
```

安装 1Panel：

```bash
curl -sSL https://resource.fit2cloud.com/1panel/package/v2/quick_start.sh -o quick_start.sh
bash quick_start.sh
```

安装完成后，终端会输出 1Panel 面板地址、用户名和密码。浏览器打开面板地址后完成初始化。

云服务器安全组建议放行：

```text
1Panel 面板端口：安装时输出的端口
HTTP：80
HTTPS：443
```

后端 `8080` 端口只给本机 OpenResty 反代使用，不建议公网放行。

## 4. 在 1Panel 安装服务

进入 1Panel 后安装：

1. `OpenResty`
2. `MySQL` 或 `MariaDB`
3. `Redis`

数据库和 Redis 不要直接暴露到公网。

## 5. 创建数据库

在 1Panel 数据库管理中创建：

```text
数据库名：ai_gateway
用户名：ai_gateway
密码：使用强密码
字符集：utf8mb4
排序规则：utf8mb4_unicode_ci
```

记录数据库连接地址、端口、用户名和密码。1Panel 容器应用的连接地址可能不是 `127.0.0.1`，以 1Panel 应用详情中显示的连接信息为准。

## 6. 上传构建产物到服务器

推荐服务器目录：

```bash
mkdir -p /opt/ai-getaway/frontend
```

可以使用 1Panel 文件管理上传，也可以从 Windows 用 `scp` 上传。

### 6.1 上传后端二进制

Windows PowerShell 示例：

```powershell
scp D:\python\ai-getaway\ai-getaway-linux-amd64 root@你的服务器IP:/opt/ai-getaway/ai-getaway
```

服务器上赋予执行权限：

```bash
chmod +x /opt/ai-getaway/ai-getaway
```

### 6.2 上传前端 dist

方式一：使用 1Panel 文件管理，把 `frontend/dist` 上传到：

```text
/opt/ai-getaway/frontend/dist
```

方式二：Windows PowerShell 使用 `scp`：

```powershell
scp -r D:\python\ai-getaway\frontend\dist root@你的服务器IP:/opt/ai-getaway/frontend/
```

最终服务器目录应类似：

```text
/opt/ai-getaway
  ai-getaway
  .env
  frontend/
    dist/
      index.html
      assets/
```

### 7.1 单机或主包 `.env`

如果只部署一台服务器，保持：

```env
CLUSTER_MODE=false
RUN_BACKGROUND_JOBS=true
```

如果主包服务器也作为多节点中的一个后端，建议开启集群模式：

```env
CLUSTER_MODE=true
INSTANCE_ID=backend-1
INSTANCE_ADVERTISE_URL=http://主包内网IP:5180
CLUSTER_INTERNAL_TOKEN=所有节点必须完全一样
RUN_BACKGROUND_JOBS=true
```

主包的 `PUBLIC_BASE_URL` 和 `ALLOWED_ORIGINS` 写公网统一域名。`DB_DSN` 和 `REDIS_ADDR` 写可写主库、主 Redis、VIP 或代理入口，不能写只读从库。

`INSTANCE_ADVERTISE_URL` 是当前后端节点对其他节点或主入口暴露的访问地址。单机可以写 `http://127.0.0.1:5180`；多机时必须写主包服务器能访问到的内网地址或负载均衡地址，例如 `http://10.0.0.11:5180`。

`CLUSTER_INTERNAL_TOKEN` 用于后台管理远程读取节点日志，所有主包和从包必须完全一致。建议生成方式：

```bash
openssl rand -base64 48
```

### 7.2 从包 `.env`

从包服务器的 `.env` 与主包保持同一套业务配置，尤其是 `JWT_SECRET` 必须完全一致。示例：

```env
APP_ENV=production
APP_PORT=5180
DB_DSN=ai_gateway:数据库密码@tcp(主库或数据库VIP:3306)/ai_gateway?charset=utf8mb4&parseTime=True&loc=Asia%2FShanghai
REDIS_ADDR=主Redis或RedisVIP:6379
REDIS_DB=0
REDIS_PASSWORD=Redis密码
JWT_SECRET=必须和主包完全一样
DEFAULT_ADMIN_EMAIL=admin@your-domain.com
DEFAULT_ADMIN_PASSWORD=只首次初始化会用到
UPSTREAM_TIMEOUT=120s
PUBLIC_BASE_URL=https://your-domain.com
ALLOWED_ORIGINS=https://your-domain.com
CLUSTER_MODE=true
INSTANCE_ID=backend-2
INSTANCE_ADVERTISE_URL=http://从包内网IP:5180
CLUSTER_INTERNAL_TOKEN=必须和主包完全一样
RUN_BACKGROUND_JOBS=true
```

多台从包只需要改 `INSTANCE_ID` 和 `INSTANCE_ADVERTISE_URL`，例如 `backend-3`、`backend-4`。`RUN_BACKGROUND_JOBS=true` 可以所有节点都开，程序会用 Redis 分布式锁保证同一类后台任务同一时间只在一个节点执行。

如果你想手工指定只有主包跑后台任务，也可以：

```env
# 主包
RUN_BACKGROUND_JOBS=true

# 从包
RUN_BACKGROUND_JOBS=false
```

这种方式更简单，但主包服务器挂掉时后台任务不会自动接手；推荐集群模式下所有节点都设为 `true`。

## 8. 创建 systemd 后端服务

```bash
vim /etc/systemd/system/ai-getaway.service
```

写入：

```ini
[Unit]
Description=AI Getaway Backend
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/ai-getaway
ExecStart=/opt/ai-getaway/ai-getaway
Restart=always
RestartSec=5
EnvironmentFile=/opt/ai-getaway/.env

[Install]
WantedBy=multi-user.target
```

启动：

```bash
systemctl daemon-reload
systemctl enable --now ai-getaway
systemctl status ai-getaway
```

查看日志：

```bash
journalctl -u ai-getaway -f
```

健康检查：

```bash
curl http://127.0.0.1:5180/health
```

期望返回：

```json
{"status":"ok"}
```

就绪检查：

```bash
curl http://127.0.0.1:5180/ready
```

单机模式下 `/ready` 至少检查数据库；集群模式下还会检查 Redis。建议负载均衡使用 `/ready` 判断后端节点是否可以接收流量。

首次启动会自动迁移数据库表，并创建 `.env` 中配置的默认管理员账号。

### 8.1 部署主包

主包服务器执行本节的 systemd 配置，并继续执行后面的 1Panel 网站、OpenResty、HTTPS 配置。主包包含：

```text
/opt/ai-getaway/ai-getaway
/opt/ai-getaway/.env
/opt/ai-getaway/frontend/dist/
```

主包适合承担统一入口：

```text
https://your-domain.com
```

如果当前只有一台服务器，主包 `.env` 使用：

```env
CLUSTER_MODE=false
INSTANCE_ID=main-1
RUN_BACKGROUND_JOBS=true
```

如果还有从包服务器一起接流量，主包 `.env` 使用：

```env
CLUSTER_MODE=true
INSTANCE_ID=backend-1
RUN_BACKGROUND_JOBS=true
```

### 8.2 部署从包

从包服务器也执行本节的 systemd 配置，但不需要执行后面的 1Panel 网站和前端静态站点步骤。从包只需要：

```text
/opt/ai-getaway/ai-getaway
/opt/ai-getaway/.env
```

从包启动后检查：

```bash
curl http://127.0.0.1:5180/ready
```

期望返回 `status` 为 `ready`。如果返回 `not_ready`，优先检查从包服务器是否能连接主库/VIP 和主 Redis/VIP。

## 9. 在 1Panel 创建网站

进入 1Panel：

1. 打开 `网站`。
2. 新建网站。
3. 选择静态网站。
4. 网站目录填写：

```text
/opt/ai-getaway/frontend/dist
```

5. 域名填写：

```text
your-domain.com
```

6. PHP 不需要开启。

## 10. 配置 OpenResty 反向代理

在 1Panel 网站配置中编辑 OpenResty/Nginx 配置，加入：

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:5180/api/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location /v1/ {
    proxy_pass http://127.0.0.1:5180/v1/;
    proxy_http_version 1.1;
    proxy_buffering off;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location = /messages {
    proxy_pass http://127.0.0.1:5180/messages;
    proxy_http_version 1.1;
    proxy_buffering off;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location / {
    try_files $uri $uri/ /index.html;
}
```

保存后在 1Panel 中重载 OpenResty。

Claude Code 常规配置 `ANTHROPIC_BASE_URL=https://your-domain.com` 时，请求通常会走 `/v1/messages`，现有 `location /v1/` 已覆盖。这里额外增加 `location = /messages`，用于兼容直接请求根级 `/messages` 的 Claude Messages API 客户端；如果站点需要提供 Claude 中转，建议保留该反代。

`location /` 的 `try_files` 很重要，否则刷新 `/admin`、`/console`、`/models` 等前端路由会 404。

### 10.1 主包 + 从包的 OpenResty 转发

如果已经部署了从包服务器，需要把 `/api/`、`/v1/`、`/messages` 转发到后端节点池，而不是只转发到 `127.0.0.1:5180`。

示例后端节点：

```text
主包后端：127.0.0.1:5180
从包后端：10.0.0.12:5180
从包后端：10.0.0.13:5180
```

OpenResty/Nginx 可使用：

```nginx
upstream ai_getaway_backend {
    server 127.0.0.1:5180 max_fails=2 fail_timeout=10s;
    server 10.0.0.12:5180 max_fails=2 fail_timeout=10s;
    server 10.0.0.13:5180 max_fails=2 fail_timeout=10s;
}

location /api/ {
    proxy_pass http://ai_getaway_backend/api/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location /v1/ {
    proxy_pass http://ai_getaway_backend/v1/;
    proxy_http_version 1.1;
    proxy_buffering off;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location = /messages {
    proxy_pass http://ai_getaway_backend/messages;
    proxy_http_version 1.1;
    proxy_buffering off;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

如果 1Panel 的站点配置不允许直接写 `upstream`，可以在 1Panel 的 OpenResty 主配置或单独 include 文件中增加 `upstream`，站点配置里只保留 `location`。也可以使用云负载均衡，把健康检查地址设置为每个后端节点的 `/ready`。

从包服务器的 `5180` 端口不要全网开放，只允许主包服务器、负载均衡或内网访问。

## 11. 配置 HTTPS

在 1Panel 网站页面：

1. 打开目标网站。
2. 进入 `SSL`。
3. 使用 Let's Encrypt 申请证书。
4. 开启强制 HTTPS。

证书生效后，确认服务器 `.env` 中：

```env
PUBLIC_BASE_URL=https://your-domain.com
ALLOWED_ORIGINS=https://your-domain.com
```

然后重启后端：

```bash
systemctl restart ai-getaway
```

## 12. 验证部署

服务器本机检查：

```bash
curl http://127.0.0.1:5180/health
```

公网 API 检查：

```bash
curl https://your-domain.com/api/settings/public
```

多后端部署时，分别在每台后端服务器本机检查：

```bash
curl http://127.0.0.1:5180/ready
```

并从主包服务器检查从包内网地址：

```bash
curl http://10.0.0.12:5180/ready
```

登录管理后台后，打开 `负载均衡` 页面，可以看到 Redis 自动发现的主包和从包节点。该页面会显示：

1. 节点在线/异常/离线状态。
2. `/ready` 通信延迟和数据库、Redis 就绪检查结果。
3. 当前节点是否执行后台任务。
4. 远程读取从包最近运行日志。

如果从包没有出现在列表中，检查从包是否与主包使用同一个 Redis、`CLUSTER_MODE=true` 是否开启、`INSTANCE_ID` 是否唯一。若节点出现但日志读取失败，检查 `INSTANCE_ADVERTISE_URL` 是否能从主包访问，以及 `CLUSTER_INTERNAL_TOKEN` 是否所有节点一致。

浏览器打开：

```text
https://your-domain.com
```

后台地址：

```text
https://your-domain.com/admin
```

使用 `.env` 中的：

```text
DEFAULT_ADMIN_EMAIL
DEFAULT_ADMIN_PASSWORD
```

登录后台。

## 16. 常见问题

### 后端启动失败，提示 JWT_SECRET

生产环境不允许使用默认密钥。修改：

```env
JWT_SECRET=一段强随机字符串
```

然后：

```bash
systemctl restart ai-getaway
```

### 后端连不上数据库

检查 `DB_DSN` 中数据库地址、端口、用户名和密码。1Panel 容器应用的连接地址以 1Panel 应用详情为准。

查看日志：

```bash
journalctl -u ai-getaway -n 100 --no-pager
```

### 刷新 /admin 或 /console 后 404

OpenResty 配置缺少前端路由回退。确认有：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

### SSE 或流式输出中断

确认 `/v1/` 代理配置包含：

```nginx
proxy_buffering off;
proxy_read_timeout 300s;
proxy_send_timeout 300s;
```

### 前端改了但服务器页面没变化

确认你已经：

1. 在 Windows 本地重新执行 `npm run build`。
2. 上传并覆盖服务器的 `/opt/ai-getaway/frontend/dist`。
3. 清理浏览器缓存或重载 OpenResty。

### 后端改了但接口没变化

确认你已经：

1. 在 Windows 本地交叉编译 Linux 二进制。
2. 上传覆盖 `/opt/ai-getaway/ai-getaway`。
3. 执行 `chmod +x /opt/ai-getaway/ai-getaway`。
4. 执行 `systemctl restart ai-getaway`。
